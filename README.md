## Traffic Pipeline Engine – Game Fix Platform

### Описание

**Traffic Pipeline Engine** — платформа для автоматического применения сетевых «фиксов» к онлайн-играм (Apex Legends, Hunt: Showdown, Roblox и др.).
Система состоит из двух частей:

- **Python-бэкенд** (`data/`) – выполняет **только мониторинг** сетевых соединений:
  - сканирует активные соединения целевых процессов;
  - выполняет обратный DNS-резолвинг (для статусов, указанных в `app_config.yaml`);
  - ведёт списки IP/доменов (используются внешним `winws.exe`);
  - записывает обнаруженные адреса в текстовые файлы (с буферизацией и автоматической записью);
  - интервал сброса буфера (`list_flush_interval`) задаётся индивидуально в каждом игровом конфиге.
- **C# WPF-лаунчер** (`PlatformLauncher/`) – графический интерфейс и вся административная логика:
  - отображает список доступных фиксов (пресетов);
  - скачивает и сохраняет конфигурации игр (YAML);
  - **добавляет/удаляет правила брандмауэра Windows** (порты TCP/UDP);
  - **управляет Cloudflare WARP** (установка, запуск, остановка, проверка статуса);
  - **создаёт и восстанавливает бэкапы** папки `lists`;
  - запускает мониторинг (в обычном режиме или только просмотр);
  - синхронизирует список пресетов с GitHub;
  - отображает вывод Python-скриптов в терминальном окне.

**Важно:** вся логика, кроме мониторинга (порты, WARP, бэкапы), вынесена в C#.
Python-скрипт `monitor.py` запускается лаунчером с нужными аргументами и только собирает статистику соединений.

---

### Запуск

Запустите `PlatformLauncher.exe` **от имени администратора**.
Лаунчер автоматически:
- проверит наличие Python 3.13+ и установит зависимости в виртуальное окружение `.venv`;
- создаст пустой `presets.yaml`, если он отсутствует;
- попытается найти папку `lists` (через процесс `winws.exe`);
- загрузит список пресетов из `presets.yaml`;
- применит настройки темы, шрифта и размера терминала из `app_config.yaml`.

### Конфигурация

#### Общий конфиг (`data/app_config.yaml`)

Содержит настройки, общие для Python и C#:

```yaml
# Общие настройки приложения
app:
  python_stop_timeout_ms: 2000   # таймаут остановки Python-процесса (мс)
  default_lists_path: ""         # путь к папке lists (если пусто — определяется автоматически)

# Настройки терминала (C# и Python)
terminal:
  theme: "Dark"                  # Dark / Light
  font_family: "Cascadia Code"   # шрифт (только C#)
  font_size: 12                  # размер шрифта (только C#)
  max_proc_width: 24             # ширина колонки процесса (Python)
  max_ip_width: 45               # ширина колонки IP (Python)
  max_port_width: 6              # ширина колонки порта (Python)
  max_domain_width: 50           # ширина колонки домена (Python)
  color_console: true            # цветной вывод (Python)
  skip_local_ips: true           # пропускать локальные IP (Python)
  highlight_style: "BRIGHT_WHITE" # стиль подсветки (Python)

# Настройки мониторинга (Python)
monitor:
  dns_resolve_statuses:          # статусы, для которых выполняется DNS-разрешение
    - "SYN_SENT"
    # - "SYN_RECV"
    # - "ESTABLISHED"
    # - "FIN_WAIT1"
    # - "FIN_WAIT2"
    # - "TIME_WAIT"
    # - "CLOSE"
    # - "CLOSE_WAIT"
    # - "LAST_ACK"
    # - "LISTEN"
    # - "CLOSING"
    # - "NONE"

# Настройки логирования (C# и Python)
logging:
  level: "INFO"
  max_file_size: 1048576         # 1 МБ
  backup_count: 5
```

#### Игровые конфиги (`data/configs/<game_id>.yaml`)

Содержат только специфичные для игры настройки:
- `target_processes` – процессы для мониторинга.
- `cloudflare_domains`, `include_domains`, `exclude_domains` – доменные списки.
- `include_ips`, `exclude_ips` – IP-списки.
- `ports` – порты TCP/UDP для брандмауэра.
- `lists` – имена файлов списков.
- `scan_interval`, `logged_connections_max`, `dns_timeout` – технические параметры.
- `list_rules` – правила добавления в списки по статусам соединений.
- `list_flush_interval` – интервал сброса буфера (по умолчанию 5.0 сек).

Все общие настройки (цвет, отступы, DNS-статусы) вынесены в `app_config.yaml`.

> ⚠️ **Известное ограничение текущей версии:** списки `include_domains`, `exclude_domains`, `include_ips`, `exclude_ips`, `pass_ips` из игрового конфига **не применяются** в Python-мониторе (ошибка загрузки). Вместо них используются пустые списки из `monitor.yaml`. Это будет исправлено в ближайшем обновлении.

### Основные возможности интерфейса

#### Вкладка «Фиксы»
- **Список игр** – отображает доступные фиксы с фильтрацией (только установленные / только локальные).
- **Установка** – скачивает YAML-конфиг игры с GitHub и применяет правила портов (брандмауэр).
- **Удаление** – удаляет конфиг и все правила портов для игры.
- **Запуск фикса** – создаёт бэкап `lists`, запускает WARP (если включено в настройках) и стартует мониторинг (требует прав администратора).
- **Мониторинг** – запускает только просмотр соединений (без изменения списков и бэкапов, доступен без прав).
- **Свойства** – настройка WARP для конкретной игры (сохраняется в `user_settings.json`).

#### Вкладка «Сервис»
- **Запустить WARP** – устанавливает (при необходимости) и подключает Cloudflare WARP.
- **Отключить WARP** – отключает WARP.
- **Темы** – переключение светлой/тёмной темы терминала (применяется из `app_config.yaml`).

#### Дополнительно
- Кнопка **«Обновить пресеты»** – синхронизирует `presets.yaml` с удалённым репозиторием.
- Кнопка **«LISTS»** – позволяет выбрать или открыть папку `lists`.
- Кнопка **«Ввод»** – контекстное меню для выполнения команд Python (удобно для отладки).

### Добавление новой игры

1. Создайте YAML-конфиг в `data/configs/<game_id>.yaml` (см. примеры).
2. Добавьте запись в `presets.yaml` с полями:
   - `id` – уникальный идентификатор;
   - `name` – отображаемое имя;
   - `config_url` – URL для скачивания конфига (или оставьте пустым для локального);
   - `warp_supported` – поддерживает ли WARP;
   - `version` – версия конфига.
3. После синхронизации (кнопка «Обновить пресеты») игра появится в списке.

### Команды Python (для разработчиков)

`monitor.py` поддерживает следующие аргументы (используются лаунчером):

- `--game <id>` – идентификатор игры.
- `--lists-path <path>` – путь к папке lists.
- `--monitor-only` – запуск в режиме только просмотра (без записи в списки).

Пример:
```bash
python data/monitor.py --game apex --lists-path "C:\path\to\lists"
```

### Логирование

- Лаунчер пишет логи в `logs/launcher_<дата>.log` (ротация по дням, до 5 файлов по 1 МБ).
- Python-монитор пишет логи в `logs/<game_id>_monitor.log` (ротация до 10 файлов по 100 КБ).
- Вывод терминала также сохраняется в `logs/console.log`.

### Примечания для разработчиков

- **Права администратора** – обязательны для запуска фиксов (изменение брандмауэра и WARP). Лаунчер перезапускается с `runas`, если прав нет.
- **WARP** – для работы требуется установленный `warp-cli`. Лаунчер автоматически скачивает и устанавливает WARP при первом запуске (через `WarpManager.InstallAsync()`).
- **Папка lists** – должна существовать и быть доступна для чтения/записи. Обычно она находится рядом с `winws.exe`. Лаунчер пытается найти её автоматически, но пользователь может указать путь вручную.
- **Синхронизация пресетов** – скачивает только `presets.yaml`. Локальный статус `installed` сохраняется.
- **Терминал** – используется `EasyWindowsTerminalControl`, который поддерживает ANSI-цвета и полную эмуляцию консоли. Тема, шрифт и размер настраиваются в `app_config.yaml`.
- **Общие настройки** – вынесены в `app_config.yaml`, игровые конфиги содержат только специфичные параметры. Это упрощает добавление новых игр и централизованное изменение внешнего вида и поведения мониторинга.

## Известные проблемы и ограничения

Текущая версия имеет следующие недостатки, которые будут исправлены в ближайших релизах:

1. **Некорректное применение фильтров из игровых конфигов**
   Списки `include_domains`, `exclude_domains`, `include_ips`, `exclude_ips`, `pass_ips` не копируются из игрового конфига в эффективный конфиг монитора. Вместо них используются пустые списки из `monitor.yaml`. Это приводит к тому, что исключения и включения не работают, и монитор обрабатывает все домены/IP без фильтрации.

2. **Отсутствие корректного graceful shutdown для Python-процесса**
   Команда `exit`, отправляемая в `stdin`, игнорируется (Python-скрипт не читает `stdin`). После таймаута процесс принудительно убивается, что может привести к потере несохранённых буферов списков и некорректному завершению работы.

3. **Функция «Починить интернет» неактивна**
   В текущей реализации все команды сброса DNS/Winsock закомментированы. Пользователь видит только текстовое сообщение, но реальных действий не происходит.

4. **Медленное добавление правил портов**
   Для каждого порта/диапазона вызывается `netsh` дважды (входящие и исходящие), что при большом количестве портов (например, в Roblox) может занимать десятки секунд.

5. **Неполная обработка ошибок в асинхронных методах**
   В конструкторе `MainViewModel` вызывается `_ = InitializeAsync()` без обработки исключений. При ошибках инициализации (например, недоступность папки `lists`) приложение может не отобразить проблему и продолжить работу в нестабильном состоянии.

6. **Утечки HANDLE в ProcessKiller**
   В циклах по процессам ресурсы не всегда освобождаются корректно (не используется `using`), что может привести к накоплению открытых дескрипторов.

7. **Отсутствие проверки существования папки lists перед запуском**
   Лаунчер не проверяет, что выбранная папка `lists` существует и содержит файлы. Если папка отсутствует, Python-скрипт завершается с ошибкой, которая не всегда корректно отображается пользователю.

8. **Пустые вкладки в разделе «Сервис»**
   Вкладки «Монитор» и «Помощь» не содержат полезной информации, что может вводить в заблуждение.

## Планы по улучшению

В следующих версиях планируется:

- 🔧 **Исправить загрузку конфигов** – явно копировать все списки из игрового конфига в эффективный.
- 🔧 **Реализовать graceful shutdown** – добавить в `monitor.py` проверку файла-флага или сигнала, чтобы завершать процесс корректно.
- 🔧 **Активировать функцию сброса DNS** – раскомментировать и реализовать вызовы с выводом результата. Изменения для Windows должны быть щадящими, без вмешательства в реестр и жестких сбросов сети.
- 🚀 **Оптимизировать добавление правил портов** – использовать PowerShell для массового добавления всех правил за один вызов.
- 🚀 **Добавить обработку исключений** во все асинхронные вызовы и отображать пользователю понятные сообщения об ошибках.
- 🚀 **Улучшить управление процессами** – переписать циклы с `using` для гарантированного освобождения ресурсов.
- 🧹 **Дополнить документацию** – добавить описание всех настроек и примеры конфигов.
